# CSS engine: `calc()` for length values

## Added

- CSS `calc()` is now evaluated for length values (previously any `calc()`
  declaration dropped). A full recursive-descent parser handles the calc type
  algebra — `length ± length`, `length × number`, `length ÷ number`, nested
  parens and precedence — and reduces the expression to a linear combination of
  a constant and viewport fractions. This lights up the app's authored forms:
  the modal `max-width: calc(100vw - 48px)` / `max-height: calc(72vh - 46px)`,
  the responsive `width: calc(100vw - 24px)`, and the row negative margins
  `margin: 0 calc(var(--sp-3) * -1)` (the `var()` resolves through the cascade,
  then folds to a constant). Those constraints now apply instead of being
  silently ignored.

## Notes

- `vw`/`vh`/`px` calc resolves to an absolute pixel `Dimension::Calc { px, vw,
  vh }` at layout time against the viewport (taffy never sees it), reusing the
  existing late viewport-unit resolution; a pure-`px` calc collapses to `Px`. A
  px-only calc on the px pathway (margins/padding) folds to a constant `f32`.
- Accepted limitation: a `calc()` that mixes a **percentage** with a length
  (`calc(100% - 14px)`) is rejected — taffy cannot represent `length + percent`
  in one value, and the app authors no such form on a supported property (the
  only one is on `background-position`, itself unsupported). `calc()` is removed
  as a blanket allowance from the `stylesheet_coverage` guardrail; a `calc()`
  that still drops must be covered by its property's `KNOWN_UNSUPPORTED` entry.
- Covered by parser unit tests (vw/vh/px forms, parens + precedence, pure-px
  collapse, the constant px-pathway fold, percent-mix rejection) and a viewport
  resolve test (`calc(100vw - 48px)` → 952px at a 1000px viewport).
