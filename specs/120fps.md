# SPEC: Sustained 120fps with Split Panes

> Status: Draft, awaiting confirmation
> Owner: Alan Martini
> Priority: P0 (top priority per user direction)
> Date: 2026-04-25
> Related code: `src/bridge.rs`, `src/pty.rs`, `src/ui/terminal_grid.rs`, `src/main.rs`, `crates/unshit-framework/crates/unshit-app/src/app.rs`, `crates/unshit-framework/crates/unshit-renderer/src/batch.rs`

## 1. Objective

Sustain 120fps in the terminal manager UI **at all times**, including when 2 or more split panes are open, when switching tabs, resizing, opening modals, or streaming output.

Today the app drops to roughly 12fps with only 2 split panes during routine tab switching. The framework already has a frame pacer aiming at the monitor refresh rate (defaults near 125fps), so the regression is in per-frame work, not the loop. We own the framework end to end, so the fix lives wherever the bug lives.

### Why this matters

A terminal that stutters during basic interaction is broken. A 120Hz pacer is meaningless if a single pane addition pushes per-frame cost over the budget. This is now P0.

### Target user

Power users running multiple shells in split panes. The product promise is "fast and snappy on a modern machine," and that promise is currently broken.

## 2. Acceptance Criteria

Acceptance bar (per user, level B): 120fps in all common scenarios, with graceful degradation only under pathological output (e.g., `yes` flooding 4 panes).

### Hard gates (must pass to ship)

| Scenario | p50 frame time | p99 frame time | Notes |
|---|---|---|---|
| 1 pane idle | < 4ms | < 8ms | baseline |
| 4 panes idle | < 4ms | < 8ms | tree size scales but no work per frame |
| 4 panes, low rate output (1 line / sec each) | < 6ms | < 8ms | streaming case |
| Tab switch, 2 panes per tab | n/a | < 8ms p99 over a 1s window | the user reported repro |
| Window resize drag, 4 panes | n/a | < 8ms p99 over a 1s window | continuous resize |
| Settings modal open or close, 4 panes | n/a | < 8ms p99 over a 1s window | |
| Cursor blink, 4 panes idle | unchanged from idle | n/a | blink must not cause rebuilds |

### Soft gates (graceful degradation)

| Scenario | p50 frame time | Notes |
|---|---|---|
| 4 panes, `yes` streaming each | < 16ms (60fps min) | acceptable degradation |
| 4 panes, `cat /dev/urandom \| xxd` each | < 33ms (30fps min) | pathological |

### Reproduction

Canonical repro that must drop to ~12fps before the fix and stay above 8ms p99 after:

1. Launch app, fresh state.
2. Split once with the split-pane key. Two panes side by side.
3. Hold a tab switch key combo for 5 seconds.
4. Read frame times from the in-app FPS overlay.

If Phase 0 (baseline) shows the canonical repro already runs at 120fps, the FPS issue is not code induced and we revisit scope.

## 3. Diagnosis Summary

A pre-spec investigation flagged five compounding causes for the 10x drop. They will be confirmed or rejected by Phase 0 measurement before any fix lands. Recording them here so we know what to look for.

1. **Whole window rebuilds.** Every `RequestRebuild` reconstructs the entire UI tree across all panes. No per-pane render isolation. Cost is roughly linear in pane count. (`src/ui/terminal_grid.rs:18`, `src/main.rs:617`)
2. **Rebuild storm.** Each pane has its own PTY subscription emitting `RequestRebuild`. Cursor blink fires every 500ms unconditionally. Events are not coalesced before reaching the pacer. (`src/bridge.rs:47`, `src/bridge.rs:148`)
3. **Sync blocking IPC.** `pty_manager.write()` calls `reply_rx.recv()` from the render closure on keyboard input. That blocks the UI thread on the daemon round trip. (`src/pty.rs:279`)
4. **No spatial culling in batch building.** 4 panes equals 4x glyph and quad instances per frame regardless of which one changed. (`crates/unshit-framework/crates/unshit-renderer/src/batch.rs:407`)
5. **Mutex contention.** `SharedState` is held during tree build while PTY readers want it. (`src/main.rs:628`)

## 4. Phased Plan

The plan is gated. Each phase ends with a measurement against the canonical repro. We do not move to the next phase until the current is verified.

### Phase 0: Measurement and baseline (mandatory first step)

Goal: confirm the regression is in code, not hardware or driver. Without this, every later step is guesswork. The user explicitly flagged uncertainty here.

1. Build an in-app **FPS overlay** that displays p50 / p95 / p99 frame times and a per-stage breakdown (tree build, layout, batch build, GPU submit). Toggle with a keybinding such as `Ctrl+Shift+F`. Wire it to the existing `FrameMetrics` struct already emitted by `crates/unshit-framework/crates/unshit-app/src/app.rs:187`.
2. Run the canonical repro. Record p50 / p95 / p99 with 1, 2, and 4 panes.
3. Record the same on a known good external app (e.g., Windows Terminal, Alacritty) on the same hardware to isolate machine vs app.
4. Compare per-stage breakdowns to identify which stage(s) explode with pane count.

Exit criteria: a written measurement table for 1 / 2 / 4 pane idle and the canonical repro, with a stage-level breakdown showing the dominant cost.

### Phase 1: Eliminate forced rebuilds (cheap wins)

Goal: stop rebuilding the entire UI tree when nothing visual has changed.

1. **Throttle the cursor blink subscription** (`src/bridge.rs:148`). Today it fires `RequestRebuild` every 500ms unconditionally on the active pane. Replace with a renderer side blink phase that toggles cursor opacity inside an existing render frame, no rebuild required. If cursor blink remains tree driven, gate it on whether the cursor is currently visible AND focused.
2. **Coalesce `RequestRebuild` in the framework.** Multiple subscriptions firing inside the same pacer window must collapse to a single rebuild. Today each PTY subscription emits independently (`bridge.rs:92` documents this).
3. **Defer non critical rebuilds.** Dimension and span sync (`bridge.rs:320`) can run on the next idle frame, not synchronously.

Exit criteria: cursor blink alone produces zero `RequestRebuild` events in the trace; idle 4 pane case shows no rebuilds beyond input / output events.

### Phase 2: Async PTY writes (kill render thread blocking)

Goal: nothing in the render closure may block on IPC.

1. Replace `DaemonPty::write()` (`src/pty.rs:279`) sync `reply_rx.recv()` with a fire and forget channel send from the UI thread. The reply (or error) is delivered via subscription back to the UI.
2. Audit all other sync IPC paths in `src/pty.rs` for similar blocking patterns. None should be on the render path.
3. Keep the existing sync API for tests and non render callers, but rename it (e.g., `write_blocking`) so the render path cannot accidentally use it.

Exit criteria: a flamegraph on the canonical repro shows zero `reply_rx.recv()` time on the render thread.

### Phase 3: Per pane render isolation

Goal: changing one pane must not rebuild the others.

1. Introduce a per pane subtree dirty flag in the framework. When a pane's grid emits `RequestRebuild`, only that subtree is reconciled, restyled, relaid, and rebatched.
2. Cache the unchanged panes' tree, style, layout, and batch from the previous frame. The renderer already has `with_persistent_buffer(true)` at `src/ui/terminal_grid.rs:284`, so the GPU side largely supports this. The CPU pipeline does not.
3. Treat tab switches as a single full rebuild (acceptable; they are discrete user actions), but make the steady state cheap.

Exit criteria: the per stage breakdown shows tree build, layout, and batch build time **independent of pane count** when only one pane is dirty.

### Phase 4: Spatial culling and batch reduction

Goal: scale frame cost by visible-changed-cells, not by total cells.

1. Refine row level dirty tracking in `CellGrid` to cell level where cheap.
2. Cull off screen panes (zoomed pane, hidden tab) from batch building entirely.
3. Reduce per frame allocations in batch building (`crates/unshit-framework/crates/unshit-renderer/src/batch.rs:407`). Pool buffers, avoid `Vec::push` reallocations on hot paths.

Exit criteria: 4 panes streaming `yes` stays above 60fps p99.

### Phase 5: Mutex contention reduction

Goal: render thread never contends with PTY readers.

1. Move snapshot construction off the global `SharedState` lock; use lock free reads of per pane grid handles.
2. Audit all `SharedState` lock sites for hold time inside the render closure.

Exit criteria: lock acquisition time on the render thread is < 100us p99 in the canonical repro.

## 7. Code Style

1. **TDD is mandatory.** Every fix follows Red, Green, Refactor. Performance fixes need a benchmark that fails before the fix and passes after.
2. **No comments unless the why is non obvious.** Performance code in particular accrues stale comments; rely on benchmark names and commit messages for the why.
3. **No backward compat shims.** Sync `write()` becomes async. Cursor blink becomes renderer side. We rename, we don't keep both.
4. **Atomic commits.** One phase substep per commit, independently revertible. `feat:`, `fix:`, `perf:`, `refactor:`, `test:`, `docs:`. Per global CLAUDE.md, do NOT add Claude as co-author.
5. **No mocking PTY for perf tests.** Per existing CLAUDE.md guidance, integration tests hit a real PTY / daemon path; mocked perf is meaningless.
6. **Run `/simplify` after each green phase.**
7. **No em dashes, no dashes as punctuation** (per user's global preference).

## 8. Testing Strategy

### Unit tests

* Cursor blink: assert no `RequestRebuild` is emitted when only blink phase changes.
* Async write: assert `pty_manager.write()` returns within 100us regardless of daemon response time (use a slow mock daemon at the IPC boundary only, not at the render layer).
* Per pane dirty: assert that updating pane A's CellGrid does not invalidate pane B's cached layout.
* RequestRebuild coalescing: assert N events inside one pacer window collapse to one rebuild.

### Integration tests

* Spawn a real PTY in 2 panes, drive output, assert frame metrics stay under budget.
* Tab switch test: programmatic tab switch, assert frame time remains under p99 budget.
* Resize test: programmatic resize loop, assert no per frame allocations beyond a fixed budget.

### Benchmarks (gating)

A new `benches/frame_time.rs` (Criterion) covering the canonical repro plus the acceptance matrix. Each phase's exit criteria is a benchmark assertion. The benchmark binary launches the renderer in headless mode and drives input via the existing event loop hooks.

### Regression tests

* Phase 0 records baseline metrics into `tests/fixtures/perf_baseline.json`. The CI bench compares against it; regression beyond a threshold fails the build.
* Every fix gets a named regression bench that would have caught the original issue.

### Coverage

`cargo llvm-cov` must not regress. Performance critical code paths should be exercised by perf tests.

### Visual verification (manual)

Per CLAUDE.md, visual regressions are verified by the user. The PR checklist includes:

* [ ] User has run `cargo run --release` with 1, 2, and 4 panes idle and seen 120fps in the overlay.
* [ ] User has performed the canonical repro and seen p99 < 8ms.
* [ ] User has streamed `yes` in 4 panes and seen >= 60fps.
* [ ] User has confirmed no flicker, tearing, or input lag introduced by changes.

## 9. Boundaries

### Always do

* Measure before changing. Every phase opens with a baseline reading.
* Fix in the framework when the bug is in the framework.
* Land changes in independently revertible commits, one per phase substep.
* Add a regression bench with each fix.
* Update CLAUDE.md "Known Framework Limitations" if a fix uncovers a previously undocumented one, or move an item out when it is fixed.

### Ask first

* Before any change to the IPC protocol between UI and `unshit-ptyd`. Touching the wire format is high blast radius.
* Before deleting any of the eager PTY spawn or initial frame correction logic in `main.rs` (CLAUDE.md flags these as load bearing).
* Before merging more than one phase at a time.
* Before disabling existing perf tests in CI to make a change land.
* Before reducing render quality (subpixel AA, glyph hinting, scrollback retention) to hit a perf number.

### Never do

* Never use `--no-verify` to bypass commit hooks.
* Never add Claude as a commit co-author.
* Never claim a fix without a benchmark showing it.
* Never introduce sync IPC on the render thread.
* Never make the cursor blink, mouse cursor, or selection rectangle drive a tree rebuild.
* Never mask FPS issues by reducing visible polling rates without a measured user benefit.

## 10. Open Questions (resolve before Phase 1)

1. **In app FPS overlay UI placement.** Floating corner widget vs settings panel toggle vs both? Default is a top right floating overlay with a keybinding toggle.
2. **CI bench enforcement.** Should the regression bench fail PRs (hard gate) or just warn? Recommend hard gate after Phase 0 establishes baselines.
3. **Headless bench environment.** Frame timing is noisy on shared CI. Run the perf bench on a dedicated runner or skip in CI and run locally with a release mode entry point. Recommend local only first, automate later.
4. **Cursor blink redesign.** Is renderer side blink (no tree rebuild) acceptable visually? Risk: a renderer blink that draws over a stale grid frame could feel laggy. Validate during Phase 1.

## 11. Done Definition

The spec is complete when:

* Canonical repro shows p99 frame time below 8ms.
* All hard gates in section 2 pass on the user's hardware.
* The FPS overlay ships and is on by default for one release for telemetry.
* `benches/frame_time.rs` covers the acceptance matrix and runs locally on every change.
* CLAUDE.md is updated with the new perf invariants (no sync IPC on render thread, cursor blink is renderer side, per pane subtree isolation exists).
